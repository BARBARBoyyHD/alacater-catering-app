# Payment & Withdrawal Architecture (Pooled Funds)

## 1. Overview
Alacater uses a **Pooled Funds** model via a payment gateway (e.g., Xendit/Midtrans). Real money is stored in the gateway's account, while our database acts as the **Source of Truth (Ledger)** for merchant ownership.

## 2. The Ledger System
We track three distinct types of money for every Provider:
- **Available Balance**: Funds that have been settled and are ready for withdrawal.
- **Pending Balance**: Funds from recent payments that are currently being processed or locked for withdrawal.
- **Platform Revenue**: Our commission (fees) tracked separately.

## 3. Core Flows

### 3.1 Payment Webhook (Money In)
**DO NOT** trust the frontend. Only the Gateway Webhook can trigger a balance update.
1.  **Gateway** sends `payment.success` webhook.
2.  **Backend** validates signature/token.
3.  **Backend** calculates:
    - `Merchant Share = Amount - Platform Fee`.
4.  **Database Transaction** (Atomic):
    - Create `OrderPayment` record.
    - Update `Provider.available_balance += Merchant Share`.
    - Create `WalletTransaction` (Type: PAYMENT).

### 3.2 Withdrawal Flow (Money Out)
Uses a **State Machine** to prevent race conditions and double-spending.

| State | Action | Balance Change |
|-------|--------|----------------|
| **REQUESTED** | Merchant initiates withdrawal | `available -= X`, `pending_withdrawal += X` |
| **PROCESSING**| Backend calls Disbursement API | No balance change |
| **SUCCESS** | Gateway confirms transfer | `pending_withdrawal -= X` |
| **FAILED** | Gateway rejects / Bank error | `available += X`, `pending_withdrawal -= X` |

## 4. Database Schema Requirements

### Wallet
Tracks the current state of a merchant's funds.
- `providerId`, `availableBalance`, `pendingWithdrawal`.

### WalletTransaction
Immutable log of every single movement of money.
- `id`, `providerId`, `amount`, `type` (PAYMENT, WITHDRAWAL, REFUND, FEE), `status`, `referenceId`.

### Disbursement
Tracks the external lifecycle of a payout.
- `id`, `amount`, `gatewayReference`, `status` (PENDING, COMPLETED, FAILED), `bankDetails`.

## 5. Security Safeguards
1.  **Atomic Transactions**: Balance updates and transaction logging MUST happen in a single Prisma `$transaction`.
2.  **Minimum Settlement (T+1)**: Funds are not "Available" until the gateway settlement window passes.
3.  **Idempotency**: All webhooks must check if the `transactionId` has already been processed to prevent duplicate credits.
